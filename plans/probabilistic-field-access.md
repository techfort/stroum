# Probabilistic Field Access — `a.~"name"`

## Context

This document describes a novel language feature for Stroum: **probabilistic compilation** via fuzzy
field access. Rather than a hard compile-time error when a field name is slightly wrong, `a.~"name"`
resolves to the *closest matching field* on the struct — first by case-insensitive exact match, then
by edit-distance similarity. This is the first introduction of probabilistic interpretation in Stroum.

Primary motivating scenario: code generated by LLMs or `#derive schema` may produce field names that
differ from hand-written code by case or minor spelling. Probabilistic access makes such code resilient.

---

## Syntax

```stroum
-- Exact static field access (existing)
a.name

-- Dynamic field access by string literal (separate plan)
a."name"

-- Probabilistic field access — resolves at compile time to closest field
a.~"name"           -- matches "name" case-insensitively, then by edit distance
a.~"FirstName"      -- may resolve to first_name, firstname, or first_Name
a.~"avg_score"      -- may resolve to average, averageScore, avg, etc.
```

---

## Resolution Algorithm (compile time)

Probabilistic access is resolved entirely at compile time (not runtime). The compiler produces a
deterministic TypeScript field access after the resolution pass. The resolution proceeds in priority
order:

1. **Exact match** — field name matches the query string exactly → same as `a.name`
2. **Case-insensitive match** — lowercase of field equals lowercase of query → use that field
3. **Prefix match** — the query string is a prefix of the field name (case-insensitive)
4. **Levenshtein edit-distance** — select the field with the smallest edit distance to the query;
   if two fields tie, use the one appearing first in the struct definition
5. **No match** — if no field scores below a threshold (e.g. edit distance > ⌊len/2⌋), emit a
   compile **error** (not a warning): probabilistic access does not silently produce `undefined`

### Threshold

A match is accepted only when edit distance `d ≤ floor(queryLen / 2)`. This prevents runaway
matches (e.g. `a.~"x"` should not resolve to a completely unrelated field).

---

## Compile-Time Output

The resolved field name is substituted at compile time. The generated TypeScript is identical to
a regular field access:

```typescript
// Stroum:  student.~"averge"  →  resolves to `average`
// TypeScript:
(student).average
```

A **compile notice** (not an error) is emitted to stderr for every probabilistic access that uses
fuzzy resolution (not exact/case-insensitive-exact):

```
notice: probabilistic field access `~"averge"` resolved to `average` (edit distance 1) at scenario.stm:7:12
```

## Changes Required

### AST (`src/ast.ts`)

Extend `FieldAccessExpression` with a `probabilistic` flag (orthogonal to the existing `dynamic` flag):

```typescript
export interface FieldAccessExpression extends ASTNode {
  type: "FieldAccessExpression";
  receiver: Expression;
  field: string;           // resolved field name after probabilistic pass
  dynamic?: true;          // a."name" → bracket notation
  probabilistic?: true;    // a.~"name" → resolved at compile time, may emit notice
  queryField?: string;     // original unresolved query (for diagnostic messages)
}
```

### Lexer (`src/lexer.ts`)

Add one new token: `TILDE = "~"`.

Rule: after a `DOT`, if the next character is `~`, emit `TILDE`. The `~` character currently has no
other use in Stroum, so there is no ambiguity.

### Types (`src/types.ts`)

Add `TILDE` to the `TokenType` enum.

### Parser (`src/parser.ts`)

In `parseFieldAccessChain()`, after consuming `DOT`:

- `IDENTIFIER` → existing static field access
- `STRING` → dynamic field access (separate plan)
- `TILDE` followed by `STRING` → probabilistic field access: store the query string as `queryField`
  and set `probabilistic: true`. The `field` on the node is left empty at parse time — it is filled
  in by the validator/resolver pass.

### Validator / Resolver (`src/validator.ts`)

New pass: **probabilistic field resolver**. After standard scope resolution, iterate all
`FieldAccessExpression` nodes with `probabilistic: true`:

1. Determine the struct type of `receiver` (requires type info from inline param annotations if
   available, or falls back to any known struct whose field set is a superset).
2. Run the resolution algorithm above against the struct's fields.
3. Assign the resolved field name to `node.field`.
4. Emit a compile notice if fuzzy resolution was used.
5. Emit a compile error if no field meets the threshold.

When struct type is unknown (e.g. untyped parameter), emit a warning and defer to runtime with
bracket notation on the lowercased query (best-effort):

```typescript
(a)["queryField".toLowerCase()]   // fallback: no compile-time guarantee
```

### Transpiler (`src/transpiler.ts`)

If `node.probabilistic` is true and `node.field` was resolved by the validator: emit standard static
field access (resolved name already in `node.field`). If unresolved fallback, emit the bracket form.

### Formatter (`src/formatter.ts`)

Restore original probabilistic syntax in formatted output (round-trip safe):

```typescript
if (expr.probabilistic) {
  return `${fmtExpr(expr.receiver, depth)}.~"${expr.queryField}"`;
}
```

### LSP (`src/lsp-completion.ts`)

When the trigger sequence `~"` is detected after a `.`, offer all known fields of the resolved
struct type as completions — same pool as static dot completions, but wrapped in the `~"..."` form.
As the user types inside the quotes, rank completions by edit distance to the current partial string.

## Probabilistic Compilation — Philosophy

`a.~"name"` introduces **probabilistic interpretation** as a first-class language concept: the
compiler itself is allowed to make a *best-guess* decision when the user's intent is approximately
expressed. The contract is:

- The resolution is **deterministic** (same input → same output, always)
- The resolution is **auditable** (a compile notice records every non-exact match)
- The resolution can **fail loudly** (threshold exceeded → hard error, not silent wrong field)
- Round-trip formatting **preserves intent** (the `.~"..."` form is kept in formatted output, not
  silently rewritten to the resolved name)

This is distinct from dynamic/runtime fuzzy matching: probabilistic access is resolved and locked in
at compile time, producing ordinary TypeScript field access.

---

## Example

```stroum
s:Student { firstName: String, averageScore: Float }

f:show s:Student =>
  println("#{s.~"firstname"}: #{s.~"avg_score"}")
  --         ↑ resolves to firstName        ↑ resolves to averageScore
```

Compile output:
```
notice: probabilistic field access `~"firstname"` resolved to `firstName` (case-insensitive) at example.stm:3:12
notice: probabilistic field access `~"avg_score"` resolved to `averageScore` (edit distance 4) at example.stm:3:36
```

Generated TypeScript:
```typescript
async function show(s: Student): Promise<void> {
  await println(`${(s).firstName}: ${(s).averageScore}`);
}
```
